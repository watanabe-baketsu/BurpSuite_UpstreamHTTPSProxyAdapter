# Build Directory

Holds Wails v3 build assets and per-platform Taskfiles. The contents were
scaffolded by `wails3 generate build-assets` and then patched so that the
frontend toolchain uses **bun** instead of npm. iOS / Android / server-mode
subdirectories from the upstream scaffold are intentionally excluded — this
is a desktop-only application.

## Layout

```
build/
├── Taskfile.yml             # Common tasks (frontend / bindings / icons / server). Patched to use bun.
├── config.yml               # `wails3 dev` configuration (hot-reload watcher rules + dev-mode commands)
├── appicon.png              # 1024x1024 master app icon (regenerated by scripts/generate_icon.py)
├── appicon.icon/            # macOS .icon source bundle, consumed by `wails3 generate icons`
├── tray/
│   ├── tray-template.png    # 32x32 black + alpha; macOS menu-bar template (auto-coloured for light/dark)
│   └── tray-regular.png     # 32x32 colour PNG; Linux/Windows tray fallback
├── darwin/
│   ├── Taskfile.yml         # macOS build/package/sign/notarize tasks (defines build:native, create:app:bundle, sign:notarize, …)
│   ├── Info.plist           # Production .app bundle Info.plist (CFBundleIdentifier = com.watanabe-baketsu.…)
│   ├── Info.dev.plist       # Dev variant (separate bundle ID for side-by-side debug builds)
│   ├── entitlements.plist   # Hardened-runtime entitlements (allows WebKit JIT)
│   ├── icons.icns           # Generated by `wails3 generate icons`
│   └── Assets.car           # Generated by `wails3 generate icons` (xcassets bundle)
├── linux/
│   ├── Taskfile.yml         # Linux build/package tasks (build:native, create:appimage, create:deb, create:rpm, …)
│   ├── desktop              # `.desktop` file template (used by AppImage / nfpm)
│   ├── appimage/build.sh    # AppImage assembly script
│   └── nfpm/                # nfpm.yaml + maintainer scripts (deb/rpm packaging)
└── windows/
    ├── Taskfile.yml         # Windows build/package tasks (build:native, create:nsis, create:msix, sign, sign:installer, …)
    ├── icon.ico             # Multi-resolution app icon
    ├── info.json            # Windows-installer metadata (version, company)
    ├── wails.exe.manifest   # Win32 manifest for the .exe
    ├── nsis/                # NSIS installer scripts
    └── msix/                # MSIX manifest templates
```

## Common build commands

```bash
# Hot-reload dev mode
wails3 dev

# Current OS
wails3 task build              # binary only
wails3 task package            # binary + native package (.app / .deb-.rpm-.AppImage / NSIS-MSIX)

# Specific platform
wails3 task darwin:build
wails3 task darwin:package      # → bin/burp-upstream-adapter.app (ad-hoc signed)
wails3 task linux:build
wails3 task linux:create:deb    # → bin/burp-upstream-adapter_*.deb
wails3 task linux:create:appimage
wails3 task windows:build
wails3 task windows:package     # → NSIS installer

# Just regenerate icons after editing scripts/generate_icon.py
wails3 task common:generate:icons

# Just regenerate frontend bindings
wails3 task bindings
```

Output goes to `bin/` at the project root (gitignored).

## Regenerating the icons

The two paths:

1. `scripts/generate_icon.py` — produces `appicon.png` (master), `windows/icon.ico`, and the two `tray/` PNGs.
2. `wails3 task common:generate:icons` — wraps `wails3 generate icons` to produce `darwin/icons.icns` and `darwin/Assets.car` from `appicon.png` + `appicon.icon/`.

The Python script handles tray icon variants that the Wails generator does not produce; the Wails command handles platform-specific encoding (`.icns`, `.ico`) that Pillow does not handle natively. Run them in that order if you've changed the master artwork.

## macOS-specific notes

- `darwin/Taskfile.yml` includes both `darwin:sign` (Developer ID only) and `darwin:sign:notarize` (Developer ID + notarytool). The release pipeline does **not** use these directly — it runs `wails3 task darwin:package` (ad-hoc) and then re-signs / notarizes manually with the existing repository secrets in [`release-macos.yml`](../.github/workflows/release-macos.yml).
- The macOS activation policy ("Hide Dock icon" toggle) is applied at runtime via `MacOptions.ActivationPolicy` in `main.go`, **not** via an `LSUIElement` key in `Info.plist`. To opt the entire build into accessory mode regardless of the user toggle, set `LSUIElement` to `<true/>` in `Info.plist`.
- `entitlements.plist` is applied during codesign (Hardened Runtime + WebKit JIT). No additional entitlements are required for the system tray.

## Windows-specific notes

- The default release pipeline currently ships a plain `.exe` zip (`wails3 task windows:build`). To switch to an installer, change the workflow to `wails3 task windows:package` and adjust artifact paths in `release-windows.yml`.
- `nsis/` is consumed by `wails3 task windows:create:nsis`; `msix/` by `wails3 task windows:create:msix`.

## Linux-specific notes

- `wails3 task linux:build` produces a bare ELF binary (current release pipeline ships this as a tar.gz).
- `wails3 task linux:create:appimage` requires `appimagetool` on PATH.
- `wails3 task linux:create:deb` / `create:rpm` use `nfpm` under the hood (no external dependencies needed; bundled by Wails).
