# Architecture

## Overview

This application is a local HTTP proxy that chains requests to an upstream HTTPS proxy with Basic authentication. It is designed to be transparent to Burp Suite — Burp sees a normal HTTP proxy on localhost, while the adapter handles TLS and credential injection behind the scenes.

```
┌─────────────┐    ┌─────────────┐    ┌──────────────────────┐    ┌───────────┐    ┌──────────┐
│ Burp Browser │───▶│ Burp Proxy  │───▶│ This Adapter (:18080)│───▶│ HTTPS     │───▶│ Internet │
│              │    │ (MITM)      │    │ - TLS connect        │    │ Squid     │    │          │
│              │    │             │    │ - Basic auth inject  │    │ Proxy     │    │          │
└─────────────┘    └─────────────┘    └──────────────────────┘    └───────────┘    └──────────┘
                         │                       │
                    HTTP history           No MITM here
                    captured here          (byte relay only)
```

## Technology Stack

| Layer | Technology |

## Data Flow

### CONNECT Tunnel (HTTPS sites)

This is the primary path for all HTTPS traffic from Burp.

```
1. Burp sends:     CONNECT example.com:443 HTTP/1.1
2. Adapter:        TLS-connect to upstream proxy
3. Adapter sends:  CONNECT example.com:443 HTTP/1.1
                   Proxy-Authorization: Basic <base64>
4. Upstream:       HTTP/1.1 200 Connection Established
5. Adapter:        HTTP/1.1 200 Connection Established  →  Burp
6. Bidirectional:  Burp ←─ io.Copy ─→ Upstream
                   (encrypted bytes, adapter doesn't inspect)
```

Key implementation details:
- Uses `http.Hijacker` to take over the raw TCP connection from Go's HTTP server
- Bidirectional relay via two `io.Copy` goroutines with `sync.WaitGroup`
- `CloseWrite()` for clean half-close when one side finishes
- Bytes transferred are tracked in atomic metrics

### Plain HTTP Forwarding

For non-HTTPS requests (rare with Burp, but supported).

```
1. Burp sends:     GET http://example.com/ HTTP/1.1
2. Adapter:        Clone request, clear RequestURI
                   Remove hop-by-hop headers
                   Set Proxy-Authorization header
3. Adapter:        http.Transport.RoundTrip() with Proxy = https://upstream
4. Response:       Remove hop-by-hop headers, copy to Burp
```

## State Management

### Backend State

| State | Storage | Lifecycle |
|-------|---------|-----------|
| Config (non-secret) | `~/.burp-upstream-adapter/adapter.config.json` (profile-aware schema, see [configuration.md](configuration.md)) | Persists across restarts |
| Active profile name | `Config.ActiveProfile` (in-memory) + persisted in the same JSON file | Selected profile drives every Start/diagnostic operation; switching is gated on the proxy being stopped |
| Password | OS keychain (account = `<profile>:<username>`) | Persists across restarts. Migrated on profile rename/duplicate. |
| Proxy server | `adapter.Server` struct | In-memory, created on Start. `App` retains the pointer; restarts always allocate a fresh `Server` + fresh `Metrics`. |
| Metrics | `adapter.Metrics` (atomics: `ActiveConnections`, `TotalRequests`, `BytesIn`, `BytesOut`, `LastError`, `LastErrorAt`) | In-memory, reset on each Start because a new `Server` is built |
| Logs | `logging.Logger` ring buffer (1000 entries) | In-memory, reset on app restart |

### Frontend ↔ Backend Communication

| Direction | Mechanism | Example |
|-----------|-----------|---------|
| Frontend → Backend | Wails v3 service bindings (auto-generated TS) | `StartProxy()`, `SaveConfig()` |
| Backend → Frontend | `app.Event.Emit(name, payload)` | `"log"` event, `"status"` event |
| Frontend → Backend (events) | `Events.Emit` from `@wailsio/runtime` (rare; Go side can subscribe via `app.Event.On`) | — |
| Backend → System tray | In-process observer callbacks (`onStatusChange` / `onProfileChange`) + 1-second metrics poll | tray label updates |
| Frontend polling | `setInterval` + bound method | `GetMetrics()` every 1 second |

## Concurrency Model

### Proxy server (`internal/adapter`)
- The proxy server runs in a separate goroutine from Wails' main loop
- Each incoming connection spawns a goroutine (via `http.Server`)
- CONNECT tunnels have two additional goroutines for bidirectional relay
- Shutdown uses `context.WithCancel` to signal all goroutines, then `http.Server.Shutdown()` for graceful drain
- Metrics use `sync/atomic` for lock-free counter updates (including `LastErrorAt` as `atomic.Int64` unix-nanos)
- The `Server.mu` mutex protects Start/Stop state transitions only

### `App` service (`app.go`)
- `App.mu` guards `cfg`, `server` pointer, and all profile mutations
- `App.obsMu` guards the tray-observer slices (`statusObservers`, `profileObservers`); never held while user code (observers) runs — observers are dispatched after copying the slice and releasing the lock
- Lock order is always `App.mu` → `Server.mu` (never inverted)
- All blocking I/O (keychain prompts, `server.Start`, `server.Stop`'s 5-second drain) happens **outside** `App.mu` so frontend RPCs and the tray ticker never stall

### System tray (`tray.go`)
- A 1-second ticker goroutine refreshes `Active connections` / `Total requests` labels and recomputes the tooltip; cancelled via `context.Context` from `BuildTray`'s returned cancel function (called via `defer` in `main.go`)
- Status changes (Start/Stop) are pushed through `onStatusChange` observers, not polled — the ticker only handles metrics and error-freshness
- All menu mutations are **in-place** (`MenuItem.SetLabel`, `SetEnabled`, plus `menu.Update()`); the tray never calls `tray.SetMenu(...)` after construction. This avoids the Windows tray crash reported in [wailsapp/wails#5227](https://github.com/wailsapp/wails/issues/5227) and the macOS click-handler loss in [#4719](https://github.com/wailsapp/wails/issues/4719)

## Security Model

1. **Password isolation**: Passwords flow from keychain → memory → auth header. Never serialized to disk or logs. Each profile's password is keyed by `<profile>:<username>` so two profiles can use the same upstream username without colliding.
2. **Loopback binding**: Default bind is `127.0.0.1`, preventing network exposure.
3. **TLS verification**: Enabled by default. Disabling requires explicit user action and shows a persistent UI warning.
4. **No MITM**: The adapter relays encrypted bytes without inspection. Burp handles MITM upstream of this adapter.
5. **Hop-by-hop cleanup**: `Proxy-Authorization` and other hop-by-hop headers are stripped from forwarded requests to prevent credential leakage to destination servers.
6. **Window-control RPC scoping**: Only methods that the frontend genuinely needs are exposed as Wails bindings. Lifecycle helpers (`onAppShutdown`, `onStatusChange`, `attachWindow`, etc.) are lowercase Go methods so the binding generator does not include them in the JS surface, keeping the RPC attack surface minimal.

## System Tray Lifecycle

```
main.go
  ├── application.New(Options{ Mac: { ActivationPolicy: <Regular | Accessory>, ApplicationShouldTerminateAfterLastWindowClosed: false }, Services: [App] })
  ├── window := app.Window.NewWithOptions(...)
  ├── window.RegisterHook(WindowClosing, …)             ← hide-to-tray when MinimizeToTrayOnClose is true
  ├── app.Event.OnApplicationEvent(Mac.ApplicationShouldHandleReopen, …)  ← Dock-click reopens window
  ├── tray := app.SystemTray.New()
  └── stopTray := BuildTray(svc, tray, window); defer stopTray()
        ├── SetTemplateIcon (macOS) / SetIcon (other)
        ├── Build menu (status / profile / start-stop / metrics / submenu / show / quit)
        ├── svc.onStatusChange  → refreshStatus(menu labels)
        ├── svc.onProfileChange → rebuild profile submenu (in place)
        └── 1-second ticker (cancellable)               ← refreshes metrics labels + tooltip
```

The macOS activation policy is decided **before** `application.New` from the user's saved `hide_dock_icon` flag, because Wails resolves the policy at native launch. Toggling the flag at runtime requires an app restart.
